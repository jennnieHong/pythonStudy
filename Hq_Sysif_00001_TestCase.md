# 일일 현금 시재관리 및 일 마감관리 (hq_sysif_00001) 테스트 케이스

본 문서는 `일일 현금 시재관리 및 일 마감관리 (hq_sysif_00001)` 화면의 EDB 마이그레이션 적합성 및 비즈니스 로직 안정성 검증을 위한 테스트 케이스입니다.

---

## 1. 로그인 및 메뉴 진입 테스트

* **목적**: 대상 화면 권한을 보유한 계정으로 로그인하여 메뉴에 정상 접근할 수 있는지 검증합니다.
* **테스트 절차**:
  1. 브라우저로 `http://localhost:8080`에 접속합니다.
  2. 이미 다른 계정으로 로그인되어 있다면 우측 상단의 **[로그아웃]** 버튼을 눌러 로그아웃합니다.
  3. 화면별 계정 권한 엑셀에 정의된 본사 사용자 계정으로 로그인합니다.
     * ID: **`7249525SHOP`** / PW: **`0000`**
  4. 좌측 내비게이션 메뉴에서 **`재고관리 > 마감관리 > 일일 현금 시재관리 및 일 마감관리`** 메뉴를 클릭하여 진입합니다.
* **기대 결과**:
  * 에러 페이지(403, 404, 500)나 Javascript 오류 없이 화면 레이아웃과 조건 검색 바가 깨끗하게 노출되어야 합니다.

---

## 2. 마감 내역 조회 기능 테스트 (getRegiList)

* **목적**: 마감 대상 일자 및 매장 검색 조건에 부합하는 일일 정산 마감 내역 목록 조회가 정상 동작하는지 확인합니다.
* **테스트 절차**:
  1. 조회 조건 바에서 매장코드를 **`NC0002` (또는 전체)**로 지정합니다.
  2. 조회 기간을 기존 샘플 데이터가 존재하는 일자 범위(예: **`2023-09-20 ~ 2023-09-30`**)로 선택합니다.
  3. **[조회]** 버튼을 클릭합니다.
* **기대 결과**:
  * EDB 데이터베이스의 `SAREGITB`(정산테이블), `IFSLRETB`(마감테이블)가 조인되어 정상적으로 마감 목록 그리드에 매장명, 일자, 마감 여부(`Y`/`N`)가 조회되어야 합니다.

---

## 3. 마감 상세 내역 조회 테스트 (getDetailRegiList)

* **목적**: 목록에서 특정 행을 선택했을 때 하단 영역에 일일 매출/입금/준비금/시재 과부족 등의 정합성 수치 상세 내역이 정상 연동되는지 검증합니다.
* **테스트 절차**:
  1. 위 2단계에서 검색 완료된 목록 그리드에서 임의의 행을 **더블클릭**하거나 선택합니다.
  2. 하단 상세 패널에 나타나는 수치 항목들을 확인합니다.
     * 판매금액, 취소금액, 현금순매출, 정산금액, 과부족금액
     * 정기준비금, 환불준비금, 준비금합계, 준비금환불, 준비금반환, 익일준비금, 입금전총액
* **기대 결과**:
  * 선택된 매장/일자의 매출 요약 값들이 공백 변환 에러 없이 정확하게 집계되어 출력되어야 합니다.

---

## 4. 마감 입금 정보 저장 테스트 (saveRegiData)

* **목적**: 본사 관리자가 확인 후 입력하는 마감입금금액 및 준비금반환액 등의 변경 사항이 DB에 오류 없이 반영(MERGE)되는지 검증합니다.
* **테스트 절차**:
  1. 상세 내역 화면의 **`마감입금`** 필드 또는 **`준비금반환`** 필드의 금액을 임의의 값(예: `150,000`원)으로 수정 입력합니다.
  2. 우측 상단의 **[저장]** 버튼을 클릭합니다.
* **기대 결과**:
  * "저장이 완료되었습니다." 알림 팝업이 발생해야 합니다.
  * DB에 접속하여 `hmsfns.IFSLRETB` 테이블의 해당 매장/일자 레코드의 `CL_DEPOSIT_AMT`, `RE_RESERVE_AMT` 값이 수정된 값으로 정상 저장/업데이트되어 있어야 합니다.

---

## 5. 현장 마감 완료 및 정합성 검증 테스트 (closeRegi)

* **목적**: 현장 마감 처리 버튼 작동 시 호출되는 비즈니스 제약 조건(매출건수 일치 여부, 로그인 오픈 일자 일치 여부)의 적절성 및 마감 플래그 변경이 정상 처리되는지 검증합니다.
* **테스트 절차**:
  1. 상세 화면 하단의 **[현장마감]** 또는 **[마감확정]** 버튼을 클릭합니다.
  2. 시스템이 내부적으로 `checkBillCnt`(영수건수 불일치 체크) 및 `checkLoginCnt`(개설포스 체크) 검증을 수행하는지 확인합니다.
     * **오류 시나리오 A**: POS 매출 건수(`STRNHDTB`)와 정산 등록 건수(`SAREGITB`)가 일치하지 않는 경우 ➔ "매출건수가 일치하지 않습니다." 알림 발생 확인.
     * **오류 시나리오 B**: 개설 정보(`OPNPOSTB`)에 등록되지 않은 포스가 있는 경우 ➔ "로그인정보와 일치하지 않습니다." 알림 발생 확인.
     * **정상 시나리오**: 모든 유효성 조건 통과 시 ➔ 마감 여부가 **`Y`**로 업데이트 확인.
* **기대 결과**:
  * 각 조건에 따른 제약 메시지가 정확히 노출되어 오작동을 차단해야 하며, 조건 충족 시 `IFSLRETB` 테이블의 `MS_CLOSE_YN` 컬럼 값이 **`Y`**로 갱신되어야 합니다.
