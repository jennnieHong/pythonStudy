# St_Stock_00002 (매장 조정 현황) QA 및 테스트 리포트

> **DB 트리거 영향도**: 없음 (단순 조회 화면이거나 트리거가 없는 테이블만 조작함)

## 1. 개요
* **대상 화면**: `[ST] 재고관리 > 조정 현황` (`st_stock_00002`)
* **테스트 일시**: 2026-05-27
* **테스트 대상 환경**: Local Tomcat (Port 8080) / EPAS DB 연동 환경
* **테스트 계정**: `fnbcafe` (비밀번호: `0000`) - 매장용 권한 (HMS SHOP CAFE)

## 2. 데이터베이스 Call-Chain 분석 및 마이그레이션 점검
제공해주신 엑셀 문서 및 운영 서버 DDL 기반으로 엔드포인트 연쇄 영향을 분석한 결과입니다.

* **관련 엔드포인트**: `/search` (조회), `/searchDt` (상세 조회)
* **직접 영향 테이블**: `IMREALTB`(실사/조정내역), `MGOODSTB`(매장상품), `TGOODSTB`(본사상품), `MMEMBSTB`, `MPRICETB` 등.
* **DB 트리거 및 프로시저 영향도**:
  * 조회(SELECT) 화면이므로 Insert/Update/Delete 시 발생하는 데이터베이스 Trigger(`IMREAL_T01`, `MGOODS_T01` 등) 및 함수 연쇄 업데이트(Cascade)는 이 화면에서 **발동하지 않습니다.**
  * 기존 Oracle에서 사용하던 함수(`F_GET_MPRICE`, `F_GET_TPRICE` 등)는 현재 MyBatis Mapper(`St_Stock_00002_Sql.xml`) 내에서 **서브쿼리 형태로 정상적으로 변환(대체)되어 코드베이스에 적용**된 것을 확인했습니다.
* **마이그레이션 검증**: CUD 로직이 아닌 조회 로직이므로, DB 트리거 없이 Mapper 쿼리가 단독으로 실행되며 기존 운영 환경과 동일한 쿼리 조회 결과를 보장하는 안전한 코드로 전환되었습니다.

브라우저 서브에이전트를 통해 `fnbcafe` 계정으로 매장 화면에 접근하여 기능이 정상 작동하는지 검증했습니다.

## 4. 최종 결론
기존 Oracle 트리거와 함수에 의존하던 부분 중 화면 조회 시 필요한 단가/원가 정보는 MyBatis의 인라인 쿼리로 완벽하게 전환되었습니다. 화면 UI와 백엔드 통신 과정에서도 데이터베이스 연결이나 드라이버 설정 오류 없이 모든 기능이 기획 의도대로 동작합니다. **(테스트 합격 - PASS)**
