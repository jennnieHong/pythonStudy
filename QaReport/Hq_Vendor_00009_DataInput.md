## 1. 데이터 라이프사이클 개요

본사 무발주 입고 등록/취소(`hq_vendor_00009`) 화면은 본사(HQ) 관리자가 개별 매장의 주문서(발주) 없이 거래처로부터 매장으로 상품이 직납된 내역을 본사 시스템에서 대리 입력하거나 관리하기 위한 **DML(CUD) 지원 화면**입니다.

이 화면은 매장 화면인 `st_vendor_00006`과 기능적으로 매우 유사하나, 본사 관리자가 체인 내의 **모든 매장을 선택하여 전표를 등록 및 제어할 수 있는 광범위한 권한**을 가집니다. 등록된 무발주 전표 정보는 EDB DB의 헤더(`OBSLPHTB`) 및 상세(`OBSLPDTB`) 테이블에 적재되며, 최종 확정 시 Java Trigger 및 코드베이스(MyBatis/Java) 수불 서비스가 연쇄적으로 작동하여 대상 매장의 수불 이력을 적재하고, 이후 야간 배치를 거쳐 현재고(`IMCRIOTB`) 및 매입 원장(`OBSLPLOG`)에 실시간 동기화 반영됩니다.

```
[본사 관리자 입력 (매장 선택 가능)]

            ┌──────────────┐      ┌──────────────┐

            └──────────────┘      └──────────────┘

                   │                     ▼

                   │                     │
                   │                     ├─▶ [수불 대기 로그 적재] Java 수불 서비스 (Sp_SUB_IMTRLG_I_Service)
                   │                     │    └─▶ 수불장(IMTRLGTB) 인서트 (배치 반영 대기)
```

## 2. 데이터 유입 및 입수 경로 (Data Input Channels)

본사 무발주 입고 데이터는 본사 백오피스 웹 어플리케이션을 통해 유입됩니다.

### 2.1 본사 웹 직접 대리 등록 채널 (Web Input)
* **입수 채널 및 화면**: 본사 백오피스 > 매입관리 > 매입입고관리 > 무발주입고등록/취소 (`hq_vendor_00009`)
* **사용자 권한**: 본사 통합 관리자 권한 (`shopadmin` 등, `systemType = 'HQ'`, `chainHqYn = 'Y'`)
* **데이터 입력 범위**: 
  * 로그인 세션의 체인 번호(`CHAIN_NO`)에 소속된 모든 하위 매장(`MS_NO`)을 선택 상자(Select Box)에서 동적으로 조회하여 매장별로 전표를 대신 등록해줄 수 있습니다.
  * 화면 상에서 입고등록 모달을 열면 **매장 선택 드롭박스**가 활성화되어 타겟 점포를 지정할 수 있는 것이 매장 전용 화면과의 핵심적인 차이점입니다.

### 2.2 핵심 컬럼 상수 및 채번 규칙
* **`CREATE_FG` (생성 주체 구분)**: 본사 무발주 등록 모듈을 통해 입력된 전표는 헤더 테이블(`OBSLPHTB`)의 `CREATE_FG` 컬럼에 `'0'` (본사 등록)으로 적재됩니다. (매장 화면 등록건은 `'1'`로 적재되어 통계 분리가 가능합니다.)
* **`NO_ORDER_YN` (무발주 구분 여부)**: 본사에서도 무발주 입고임을 마킹하기 위해 `'Y'` 상수로 적재됩니다.
* **`ORDER_FG` (발주 구분)**: 무발주 입고 전표를 발주 유형 중 무발주로 매핑하기 위해 `'1'`로 고정 적재됩니다.
* **`SLIP_FG` (전표 구분)**: 무발주 입고를 구분하는 `'0'` (입고)으로 저장됩니다.
* **`SLIP_NO` (전표번호 채번)**:
  * `Hq_Vendor_00009_Mapper.selectSlipNo` 쿼리를 호출하여, 입력된 매장(`MS_NO`)과 일자(`ORDER_DATE`) 기준으로 기존 최고 전표번호를 구해 `1`을 더해 4자리 제로 패딩 문자열 형태로 채번합니다. (예: `0001`, `0002` 등)
* **`LINE_NO` (상세 순번 채번)**: 
  * `Hq_Vendor_00009_Mapper.selectMaxLineNo` 쿼리를 통해 해당 매장 및 전표 안에서 순차적 일련번호 형태로 가산되어 생성됩니다.

## 4. 데이터 정제 및 가공 규칙 (Data Cleansing Rules)

### 4.1 의제매입세액 (Fictitious VAT) 자동 계산
면세 대상이면서 의제매입세액 대상(`FICTITIOUS_YN = 'Y'`)이고 부가세 구분이 면세인 품목의 경우, 본사 수정 쿼리(`updateGoodsDtList`)에서도 매입가액에 비례한 의제세액이 실시간 자동 연산되어 저장됩니다.
* **산출식**:
  * `FICTITIOUS_VAT_AMT = ROUND((PURCH_AMT * 6) / 106)`

### 4.2 헤더 실시간 합계 동기화 (`updateGoodsHd` / `updateOrdPurchHd`)
상세 테이블(`OBSLPDTB`)의 개별 라인 단가 및 수량이 변동되면, SQL Mapper에서 자동으로 해당 전표의 상세 품목 공급가액, 부가세, 의제매입세액의 총 합계를 집계하여 헤더(`OBSLPHTB`) 테이블에 실시간 동기화(Sync) 업데이트를 처리합니다.

## 5. 실시간 재고 및 회계 연쇄 작용 (Depth 3 Trigger & Service Flow)

본사 무발주 전표 확정(`PROC_FG = '4'`) 및 취소(`PROC_FG = '5'`) DML 트랜잭션이 수행되면 코드베이스 및 EDB 데이터베이스 상에서 연쇄 작용이 수행되어 **선택된 대상 매장의 재고**와 **매입 회계 정보**를 실시간으로 가감시킵니다.

### 5.1 [1단계] Java Trigger 발동 (`Tr_OBSLPD_T01_Service`)
상세 테이블(`OBSLPDTB`)의 진행 상태가 확정 또는 취소로 갱신되면 트리거 서비스가 작동합니다.
1. **취소일 정합성 검증**: 진행 상태가 `'5'`(취소)인 경우, 전표 헤더의 취소 연월과 매입 연월이 일치하는지 검증합니다. (만약 다를 경우 `OBSLPD_T01 : CANCEL_DATE ERROR!!` 런타임 예외를 발생시키며 트랜잭션을 전체 롤백시킵니다.)
2. **낱개 기준 환산 수량 연산**: 수불 처리를 위해 수량에 낱개 입수수량을 곱해 최종 낱개 단위 수량을 계산합니다. (`qty = purchInQty * invInQty`)
3. **대체 분기 판별**: 상품 마스터(`TGOODSTB`)의 세트 상품 구분(`SET_FG`)에 따라 대상 재고 처리 Java 수불 서비스로 자동 분기 처리됩니다.

### 5.2 [2단계] Java 수불 서비스 실행 (`Sp_SUB_IMTRLG_I_Service`)
일반 상품의 경우 코드베이스로 전환된 `Sp_SUB_IMTRLG_I_Service` 가동을 통해 해당 전표에 명시된 **대상 매장의 재고 원장**을 조작합니다.
* **수불 원장 기록 (`IMTRLGTB` 적재)**:
  * 입고 확정 시: 매입 전표 데이터에 매핑된 수량과 코스트가 양수(+)로 신규 인서트되어 `IMTRLGTB` 테이블에 `PROC_YN = 'N'` 상태로 기록되어 배치 처리를 대기합니다.
  * 입고 취소 시: 기존 확정 정보를 연계 키(`KEY_BILL_NO`)로 활용하여 수량 및 코스트가 마이너스(-) 음수로 적재됩니다.
* **현재고 및 원장 배치 반영**:
  * 실제 대상 매장의 현재고 원장 테이블(`IMCRIOTB`)의 현재고(`CUR_QTY`) 및 일/월수불 원장(`IMDDIOTB`/`IMMMIOTB`)은 야간 또는 주기적으로 구동되는 재고 반영 배치 프로그램(`DmIMTR01Service`)에 의해 최종 합산 가감산(`CUR_QTY = CUR_QTY + qty`)됩니다.

### 5.3 [3단계] 본사/매장 매입 이력 로그 적재 트리거 (`Tr_OBSLPH_T01_Service`)
헤더 테이블(`OBSLPHTB`)의 상태가 갱신 완료되면 작동하여 매입 정산의 기초가 되는 `hmsfns.OBSLPLOG` 테이블에 로그 데이터를 인서트합니다.
* **로그 적재 내용**:
  * 확정 시: 해당 거래처 및 대상 매장의 매입 공급가액 및 부가세를 양수(+)로 로그 원장에 적재합니다.
  * 취소 시: 정산 마감을 위해 취소 전표에 해당하는 역대체 금액을 음수(-) 마이너스로 분개 대체 적재하여 본사 차원의 마감 정합성을 유지합니다.

### 5.4 재고 반영 확인 관련 화면 및 반영 시점
본사 관리자가 `hq_vendor_00009`에서 대리 등록 및 확정한 매장 재고 변동량은 배치 프로그램에 의해 최종 반영된 이후 화면을 통해 모니터링할 수 있습니다.
* **관련 확인 화면**: **[HQ] 재고관리 > 재고현황 > 현재고조회 (`hq_stock_00001`)**
* **동작 원리**:
  * 본사 관리자가 특정 가맹점(`MS_NO`)을 지정하여 입고 확정을 처리하면, `IMTRLGTB` 수불 대기 로그로 실시간 적재됩니다.
  * 이후 재고 반영 배치 프로그램(`DmIMTR01Service`)이 실행되어 `IMTRLGTB` 로그를 읽고 대상 매장의 일수불(`IMDDIOTB`) 및 월수불(`IMMMIOTB`) 데이터를 합산 갱신합니다.
  * 본사 현재고조회(`hq_stock_00001`) 화면은 대상 매장의 월수불(`IMMMIOTB`)과 일수불(`IMDDIOTB`) 수량의 합산값(`MM_QTY + DD_QTY`)을 기준으로 현재고를 계산하여 출력하므로, **배치가 정상 구동되어 반영을 마친 시점부터 변경된 재고량이 화면에 즉시 조회**됩니다.

### 5.5 새벽배치(Batch) 연동 제외 및 비즈니스 이유
본사 무발주 화면에서 관리하는 전표(`ORDER_FG = '1'`) 역시 새벽배치의 매입 전송 인터페이스에서 제외됩니다.
* **배치 쿼리 및 프로시저 필터링 조건 (`ORDER_FG = '0'`)**:
  * 새벽 배치 전송 쿼리(`ObslInerface_SqlMapper.xml`의 `selectObslTargetList`) 및 핵심 DB 프로시저(`SUB_IF_HYDM_SEND_P`의 `CURSOR OBSL` 커서)는 오직 **`AND A.ORDER_FG = '0'`** 조건이 적용된 발주 매입 건만 수집하여 본사 회계 시스템으로 전송합니다.
* **비즈니스적 사유**:
  * **발주 입고 (`ORDER_FG = '0'`)**: 본사 주도의 정식 수발주 거래이므로 전국 매장 거래을 모아 본사 회계 시스템(ERP)에 등록하여 대금 지급 및 세무 계산을 처리(새벽배치 연동)해야 합니다.
  * **무발주 입고 (`ORDER_FG = '1'`)**: 현장 매장 단위의 긴급/즉석 직납 거래이므로 본사 회계팀에서 대금을 일괄 집행하지 않고 매장 로컬 비용이나 별도 현장 정산 수단을 타므로 배치가 불필요합니다.
