## 1. 데이터 라이프사이클 개요

매장별 입출금현황(`hq_cash_00002`) 화면은 본사(HQ)에서 개별 매장의 입출금 및 지출 실적 데이터를 통합하여 모니터링하기 위한 **조회 전용(Read-Only)** 화면입니다. 본 화면에 표출되는 데이터는 매장 현장(POS 단말기) 또는 웹 백오피스(Store Web)의 다양한 입력 채널로부터 수집 및 정제되어 관리됩니다.

```
[입력 채널: Web/POS] ────> [실적 적재: MACCIOTB] ────> [데이터 정제/필터링] ────> [HQ 조회 화면 표출]
                                 │
                        [마스터 매핑: MMACNTTB]
```

## 2. 데이터 유입 및 입수 경로 (Data Input Channels)

매장 입출금 데이터는 발생 주체 및 시점에 따라 크게 **웹 직접 등록(Web Input)**과 **포스 연동(POS Interface)**의 2가지 채널을 통해 입수됩니다. 이 구분은 `MACCIOTB` 테이블의 `CREATE_FG` 컬럼을 통해 추적 및 관리됩니다.

### 2.1 웹 직접 등록 채널 (Web Input: `CREATE_FG = '0'`)
매장 점주 또는 매니저가 점포용 백오피스 웹 애플리케이션에 로그인하여 매장 내 발생 비용 및 수동 현금 입출금 내역을 입력하는 경로입니다.

* **입수 채널 및 화면**:
  * **[ST] 입출금내역등록 (`st_cash_00001`)**: 일 단위의 소액 잡비 지출, 소모품 구매, 기타 소액 현금 입출금 내역을 관리합니다.
  * **[ST] 월간지출내역등록 (`st_cash_00004`)**: 임대료, 관리비, 세금 등 월 1회 주기적으로 발생하는 고정 및 변동 경비 지출을 일괄 관리합니다.
* **기술적 동작 메커니즘**:
  * **호출 API**: `POST /backoffice/data/st/cash/st_cash_00001/cashSave`
  * **호출 쿼리**: `St_Cash_00001_Mapper.insertCash`
  * **데이터 특징 및 적재 칼럼**:
    * **`CREATE_FG`**: 웹 등록을 의미하는 `'0'` 상수로 강제 적재됩니다.
    * **`USER_ID` 및 `CREATE_ID`/`LAST_ID`**: 로그인 세션의 사용자 ID(예: `shopadmin` 등)가 저장됩니다.
    * **`ACCIO_NO` 채번 규칙**: `00`으로 고정된 상우 2자리와 순차적 시퀀스 2자리를 병합하여 `0001`, `0002` 형태로 채번됩니다. (`SUBSTR(ACCIO_NO, 1, 2) = '00'` 조건 사용)
    * **`REMARK`**: 매장 사용자가 팝업 모달창에서 입력한 비고 설명 텍스트가 그대로 적재됩니다.

### 2.2 포스 인터페이스 채널 (POS Interface: `CREATE_FG = '1'`)
매장 영업 현장의 POS 단말기에서 발생한 준비금 등록, 돈통 개폐, 중간 마감 등의 금고 현금 시재 정산 데이터가 백그라운드 연동 데몬을 통해 동기화 유입되는 경로입니다.

* **입수 채널 및 데몬**:
  * **배치 데몬명**: `DmSALT10` (POS 입출금계정 반영 데몬)
  * **실행 SQL 맵퍼**: `DmSALT10_SQL.xml`
* **기술적 동작 및 동기화 메커니즘**:
  1. **POS 1차 유입**: 매장에서 POS 마감 또는 정산 완료 시점에 해당 전문 데이터가 인터페이스 임시 테이블(`HMSFNS.TB_IF_SA_POS_ACCOUNT_IO_CTNT`)로 실시간 적재됩니다. (이때의 진행 상태 코드 `PROC_FG = '0'`)
  2. **데몬 주기 실행**: `DmSALT10` 데몬이 주기적으로 실행되어 미처리 데이터(`PROC_FG = '0'`)를 100건씩 읽어 들입니다. (`select_TB_IF_SA_POS_ACCOUNT_IO_CTNT` 쿼리)
  3. **본 테이블 이관**: 임시 테이블의 JSON 포맷 데이터(`JOURNAL_CTNT`)를 파싱하여 메인 DB의 `HMSFNS.MACCIOTB` 테이블에 Insert 또는 Update 처리합니다.
     * 호출 쿼리: `DmSALT10_SQL.insertMACCIOTB` / `DmSALT10_SQL.updateMACCIOTB`
  4. **처리 완료 및 정리**: 이관이 완료되면 해당 데이터를 인터페이스 임시 테이블에서 삭제(`delete_TB_IF_SA_POS_ACCOUNT_IO_CTNT`)하거나 로그 이력 테이블(`TB_IF_SA_POS_ACCOUNT_IO_CTNT_LOG`)로 옮깁니다.
* **데이터 특징 및 적재 칼럼**:
  * **`CREATE_FG`**: POS 자동 연동을 의미하는 `'1'` 상수로 적재됩니다.
  * **`USER_ID` 및 `CREATE_ID`/`LAST_ID`**: 시스템 기본 연동 식별자인 `'POS_USER'` 상수로 일괄 적재됩니다.
  * **`ACCIO_NO` 채번 규칙**: `POS_NO(앞 2자리) + SEQ(뒤 2자리)` 형태로 구성되어 어느 POS 장비에서 정산된 입출금인지 식별할 수 있게 생성됩니다.
  * **`REMARK`**: POS 시스템에서 생성한 자동 로그 내용(예: "준비금 등록", "돈통 강제 개폐" 등)이 자동으로 맵핑되어 기록됩니다.

### 2.3 일반 주문/매출(Sales) 데이터와의 차이점 및 비즈니스 목적성
본 화면 및 `MACCIOTB` 테이블에서 취급하는 입출금 데이터는 일반적인 상품 판매 거래(주문 결제 시 카드/현금/복합결제 등)와는 성격과 목적이 완전히 다릅니다.

* **일반 매출 데이터와의 핵심 차이점**:
  * **주문/매출 데이터 (Sales Data)**: 고객이 매장에서 상품을 구매할 때 발생하는 실시간 판매 데이터(카드, 현금, 간편결제 등 복합결제 포함)로, 이는 판매 원장 테이블(예: `TSALESTB` 등)에 영수증 건별로 누적됩니다.
  * **입출금 실적 데이터 (`MACCIOTB`)**: 개별 판매 건이 아니라, **매장 내 현금 금고(시재)의 실제 흐름 및 점포 운영 경비**를 관리하기 위한 데이터입니다.
* **비즈니스적 기록 시점 및 용도**:
  1. **POS 시재 정산 및 마감 시점 (`CREATE_FG = '1'`)**:
     * 매장 오픈 시 영업을 위해 돈통에 채워두는 **시조금(준비금)** 등록 시점
     * 근무자 교대 또는 중간 정산 시점에 금고로 현금을 회수하는 **시송금(중간마감)** 시점
     * 돈통 오차(시재 부족/과잉)를 맞추거나 시재 조정을 위해 **돈통을 물리적으로 개폐하고 정산**하는 이벤트 시점
  2. **매장 운영 경비 집행 시점 (`CREATE_FG = '0'`)**:
     * 매장 운영에 수반되는 수도세, 전기세, 가스비 등의 **월 단위 공과금 납부** 시점
     * 매장 청소용품, 사무용품 등 현장에서 임의로 소요되는 소액 **잡비/소모품 지출**이 발생하는 시점

## 3. 핵심 데이터베이스 테이블 및 스키마

### 3.1 MACCIOTB (매장 입출금 실적)
매장에서 발생하는 모든 입금 및 출금(지출)의 거래 내역을 기록하는 거래 원장(Transaction) 테이블입니다.

| 컬럼명 | 타입 | 필수 여부 | 설명 |
|--------|------|-----------|------|
| **ACCIO_DATE** | VARCHAR(8) | PK (1) | 입출금 발생 일자 (YYYYMMDD) |
| **MS_NO** | VARCHAR(10) | PK (2) | 매장코드 |
| **ACCIO_NO** | VARCHAR(4) | PK (3) | 입출금 번호 (PosNo 2자 + Seq 2자) |
| **CHAIN_NO** | VARCHAR(10) | N | 체인번호 |
| **ACNT_FG** | VARCHAR(1) | Y | 계정구분 (0: 입금, 1: 출금) |
| **ACNT_CD** | VARCHAR(10) | Y | 계정코드 |
| **ACNT_AMT** | NUMERIC(15,2) | Y | 계정 금액 (부가세 별도) |
| **VAT** | NUMERIC(15,2) | Y | 부가세 |
| **CUST_CD** | VARCHAR(10) | N | 거래처코드 |
| **REMARK** | VARCHAR(100) | N | 비고 (특이사항 등) |
| **DELETE_YN** | VARCHAR(1) | Y | 삭제 여부 ('Y': 삭제, 'N': 정상) |
| **CREATE_FG** | VARCHAR(1) | Y | 생성주체 ('0': Web, '1': POS) |
| **USER_ID** | VARCHAR(45) | N | 최초/최종 등록자 ID |

### 3.2 MMACNTTB (가맹점 입출금 계정 마스터)
각 매장별로 사용할 수 있도록 허용된 입출금 계정 과목(명칭)을 관리하는 기준 정보(Master) 테이블입니다.

| 컬럼명 | 타입 | 필수 여부 | 설명 |
|--------|------|-----------|------|
| **MS_NO** | VARCHAR(10) | PK (1) | 매장코드 |
| **ACNT_FG** | VARCHAR(1) | PK (2) | 계정구분 (0: 입금, 1: 출금) |
| **ACNT_CD** | VARCHAR(10) | PK (3) | 계정코드 |
| **ACNT_NM** | VARCHAR(100) | Y | 계정 과목 명칭 (예: 기타입금, 소모품비 등) |

## 4. 데이터 정제 및 가공 규칙 (Data Cleansing Rules)

최종 화면인 `hq_cash_00002`에 데이터를 뿌려주기 전에 SQL Mapper 레벨에서 아래와 같은 규칙으로 데이터를 실시간 정제 및 집계합니다.

### 4.1 매장 선택 드롭다운 리스트(Select Box) 데이터 입수 및 필터링 기준
매장 조회 조건 영역의 매장 선택 드롭박스는 공통 모듈 API를 통해 데이터를 로드하며, 권한 및 소속에 따라 동적으로 필터링됩니다.
* **호출 API**: `POST /backoffice/data/common/condition/selectMsPos`
* **실행 SQL**: `CommonModule_GoodsClass_Sql.xml - selectMemberList`
* **기준 정보 및 조인 규칙**:
  * **조회 대상**: 매장 마스터 테이블(`hmsfns.MMEMBSTB`)
  * **매장명 조합 규칙**: `[영업상태 명판] + 매장명` 포맷으로 제공됩니다.
    * 영업상태 공통코드 명판 테이블(`hmsfns.MNAMEMTB`)에서 영업상태 구분 코드(`NM_FG = '130'`)와 매장 마스터의 `OPEN_FG` 코드를 조인하여 명판 명칭(예: `[오픈]`, `[폐점]`)을 추출해 매장 명칭 앞에 대괄호 포맷으로 바인딩합니다.
* **권한 및 세션별 필터링 규칙**:
  * **소속 체인 필터**: 세션의 체인 번호(`CHAIN_NO`)와 일치하는 매장만 조회합니다.
  * **본사 권한 (`chainHqYn = 'Y'`)**: 체인 하위의 전체 매장 목록이 드롭박스에 노출됩니다.
  * **가맹점 권한 (`chainHqYn = 'N'`)**: 본인의 매장 정보(`MS_NO = #{userMsNo}`)만 1건 노출되도록 강제 제한됩니다.
  * **체인 구분 필터 (`chainFg` 파라미터가 없거나 공백일 시)**: 본사 점포를 제외하고 일반 가맹점만 드롭박스에 표출되도록 `CHAIN_HQ_YN = 'N'` 조건이 기본 적용됩니다.

### 4.2 계정구분(ACNT_FG) 코드값 및 옵션 정의
매장별 입출금 현황 및 실적 원장에서 계정의 유형은 `ACNT_FG` 필드를 통해 분류됩니다. 현재 정의된 구분 값 및 화면상의 매핑 규칙은 다음과 같습니다.
* **데이터베이스 및 코드값 정의**:
  * `0`: **입금** (Deposit / Inflow) - 매장에 현금이 유입되거나 시조금 등록, 기타 입금이 발생한 경우.
  * `1`: **출금** (Withdrawal / Outflow / 지출) - 잡비 지출, 고정 지출, 물품 대금 송금 등 매장에서 현금이 지출된 경우.
* **화면 렌더링 포맷팅 (Formatter)**:
  * Bootstrap Table 그리드 바인딩 시 `acntFgFormatter`를 통해 가독성이 개선된 포맷으로 변환 출력됩니다.
    * `0` -> `[0] 입금`
    * `1` -> `[1] 출금`
* **차트 연계**:
  * 그리드 데이터 로드 완료 이벤트(`load-success.bs.table`) 시점에 `ACNT_FG` 값에 따라 각기 다른 성격의 차트 객체로 분기하여 데이터를 매핑합니다.
    * `ACNT_FG == '0'`: 좌측 `hq_cash_00002_chart01` (입금 계정별 매출현황)에 바인딩
    * `ACNT_FG == '1'`: 우측 `hq_cash_00002_chart02` (출금 계정별 매출현황)에 바인딩

### 4.3 데이터 가공 및 집계 규칙
1. **유효 데이터 필터링 (Cleansing)**:
   * `DELETE_YN = 'N'`: 화면이나 API를 통해 삭제 처리된 실적은 집계 및 요약 대상에서 완전히 제외합니다.
   * `ACNT_FG IN ('0', '1')`:
     * **왜 (Why)**: 테이블 스키마상 `ACNT_FG`가 `VARCHAR(1)`로 되어 있어 Check Constraint가 부재할 경우 데이터 수동 조작이나 시스템 에러로 쓰레기 데이터(Null, 공백, 타 문자 등)가 유입될 수 있습니다. 이를 미연에 방지하여 순수 입금(`0`) 및 출금(`1`) 합계 금액의 정합성을 보장합니다. 또한 계정 마스터(`MMACNTTB`)와 조인 시 비정상 키로 인한 조인 에러 및 실적 누락을 방지하기 위한 방어적 필터링 조건입니다.
     * **언제 (When)**: 본사 화면(`hq_cash_00002`)에서 점포별 실적을 집계하는 `selectMmaList`(메인 그리드)와 세부 내역을 바인딩하는 `selectDetailMMaList`(상세 모달) 쿼리가 호출되어 실행되는 실시간 조회 시점마다 매번 적용됩니다.
2. **마스터-트랜잭션 데이터 매핑 (Enrichment)**:
   * `MACCIOTB` 테이블의 `ACNT_FG`, `ACNT_CD`, `MS_NO` 값을 기준 정보인 `MMACNTTB`와 INNER JOIN하여, 규격화된 계정 명칭(`ACNT_NM`)을 매핑합니다.
3. **기간 집계 및 요약 (Aggregation)**:
   * 메인 화면의 그리드(`hq_cash_00002_t01`) 조회를 위해 실적 데이터를 계정별로 통합 연산합니다:
     * 집계 공식: `SUM(IO.ACNT_AMT) AS ACNT_AMT`
     * 정제 기준: `GROUP BY IO.ACNT_FG, IO.ACNT_CD, NT.ACNT_NM`
4. **상세 원장 데이터 포맷팅 (Formatting)**:
   * 상세 모달(`hq_cash_00002_M01`) 내 상세 그리드(`hq_cash_00002_t02`) 바인딩 시 날짜 문자열 포맷팅을 수행합니다:
     * `YYYYMMDD` 문자열을 `YYYY-MM-DD` 형태의 표준 날짜 형식으로 치환하여 가독성을 높입니다.

### 4.4 데이터 논리 삭제(Soft Delete) 처리 메커니즘 및 정책
실적 데이터는 데이터 보존 및 무결성을 보장하기 위해 물리적으로 완전히 지우지 않고 플래그를 변경하는 **논리 삭제(Soft Delete)** 메커니즘을 채택합니다.

* **삭제 처리 시점 및 트리거**:
  * 매장 사용자가 **[ST] 입출금내역등록 (`st_cash_00001`)** 또는 **[ST] 월간지출내역등록 (`st_cash_00004`)** 화면에서 등록된 거래 행을 선택하고 **[삭제] 버튼**을 누를 때 트리거됩니다.
  * 프론트엔드가 삭제 API(`POST /backoffice/data/st/cash/st_cash_00001/cashDel`)를 호출하면, 백엔드 서비스가 호출되어 MyBatis Mapper(`St_Cash_00001_Mapper.deleteYnUpdate`)를 실행합니다.
* **실행 SQL**:
  ```xml
  UPDATE hmsfns.MACCIOTB
     SET DELETE_YN = 'Y'
   WHERE (ACCIO_DATE, MS_NO, ACCIO_NO) IN ((#{item.accioDate}, #{item.msNo}, #{item.accioNo}))
  ```
* **논리 삭제 정책을 채택하는 이유 (Why)**:
  * **감사 이력 보존 및 오등록 복구 (Audit & Recovery)**: 근무자의 오작동이나 부주의로 내역이 지워졌을 때 역추적 및 관리자 승인 하에 간편하게 복원할 수 있도록 히스토리를 보존합니다.
  * **POS 데이터와의 정합성 불일치 예방**: POS 기기로부터 자동 인터페이스된 데이터(`CREATE_FG = '1'`)의 경우, 물리 삭제되면 POS 로컬 마감 내역과 백오피스 정산 데이터의 Primary Key(거래 거래일, 점포코드, 거래번호 등)가 끊어져 대조 정합성이 훼손될 수 있습니다.
  * **과거 마감 데이터 보호**: 집계 배치 프로그램이나 정산 보고서에 이미 합산 연산된 금액 데이터가 유실되어 과거 마감 내역이 훼손되는 것을 방지합니다.
