## 1. 데이터 라이프사이클 개요

매장 무발주 입고 등록/취소(`st_vendor_00006`) 화면은 매장(Store) 사용자가 본사 주문(발주) 과정 없이 거래처로부터 상품을 직접 인수(무발주 입고)하였을 때, 해당 입고 내역을 수동으로 등록하고 최종 확정 또는 취소 처리하는 **DML(CUD) 지원 화면**입니다.

이 화면에서 발생한 데이터는 입고 전표 헤더 테이블(`OBSLPHTB`)과 상세 테이블(`OBSLPDTB`)에 기록되며, 최종 확정 및 취소 시 Java Trigger 및 코드베이스(MyBatis/Java) 수불 서비스를 통해 실시간 재고 원장(`IMCRIOTB`/`IMTRLGTB`) 및 매입로그(`OBSLPLOG`) 테이블로 연쇄 반영됩니다.

```
[매장 사용자 입력]
        │ (등록/수정)
        ▼
 ┌──────────────┐      ┌──────────────┐
 │   OBSLPHTB   │◀────▶│   OBSLPDTB   │
 │ (무발주 헤더) │      │ (무발주 상세) │
 └──────────────┘      └──────────────┘
        │                     │
        │                     │ (확정/취소 트랜잭션 시)
        │                     ▼
        │              [Java Trigger] Tr_OBSLPD_T01_Service
        │                     │
        │                     ├─▶ [재고반영] Java 수불 서비스 (Sp_SUB_IMTRLG_I_Service)
        │                     │    └─▶ 수불장(IMTRLGTB) 인서트 & 현재고(IMCRIOTB) 실시간 가감산

```

## 2. 데이터 유입 및 입수 경로 (Data Input Channels)

매장 무발주 입고 데이터는 매장 백오피스 웹 페이지에서 사용자가 직접 입력하는 경로를 통해 단일하게 입수됩니다.

### 2.1 매장 웹 직접 등록 채널 (Web Input)
* **입수 채널 및 화면**: 매장 백오피스 > 매입관리 > 매입입고관리 > 무발주입고등록/취소 (`st_vendor_00006`)
* **사용자 권한**: 매장 관리자 권한 (`fnbcafe` 등, `systemType = 'SHOP'`)
* **데이터 입력 범위**: 로그인 세션에 속한 매장 코드(`MS_NO = #{userMsNo}`)로만 전표 작성이 제한되며, 본사 권한과 다르게 타 매장의 입고를 대리 등록할 수 없습니다.

### 2.2 핵심 컬럼 상수 및 채번 규칙
* **`CREATE_FG` (생성 주체 구분)**: 매장 웹 화면에서 등록된 무발주 입고의 경우, 헤더 테이블(`OBSLPHTB`)에 `'1'` (매장 등록) 코드로 적재됩니다. (본사 무발주는 `'0'`으로 적재되어 등록 주체가 명확히 분리됩니다.)
* **`NO_ORDER_YN` (무발주 구분 여부)**: 무발주 입고임을 명시하기 위해 항상 `'Y'` 상수로 고정 적재됩니다.
* **`ORDER_FG` (발주 구분)**: 무발주 입고 전표이지만 발주 유형 코드 중 무발주 매입을 나타내는 `'1'`로 고정 적재됩니다.
* **`SLIP_FG` (전표 구분)**: 입고 전표를 나타내는 `'0'` (입고)으로 저장됩니다.
* **`SLIP_NO` (전표번호 채번)**: 
  * `St_Vendor_00006_Mapper.selectSlipNo` 쿼리를 호출하여 해당 일자(`ORDER_DATE`)와 매장(`MS_NO`)의 기존 최대 전표번호를 구한 뒤, `1`을 가산하여 4자리 패딩 문자열 형태로 채번합니다. (예: `0001`, `0002` 등)
* **`LINE_NO` (상세 순번 채번)**: 
  * `St_Vendor_00006_Mapper.selectMaxLineNo` 쿼리를 통해 해당 전표 내에서 일련번호(1부터 시작) 형태로 자동 부여됩니다.

## 4. 데이터 정제 및 가공 규칙 (Data Cleansing Rules)

### 4.1 의제매입세액 (Fictitious VAT) 자동 계산
상세 상품 저장 및 일괄 수정 시, 해당 상품이 면세 대상이면서 의제매입세액 공제 대상(`FICTITIOUS_YN = 'Y'`)이고 전표의 부가세 구분이 면세인 경우, 매입 가액에 비례하여 법정 의제 매입 세액율(예: 6/106)에 맞게 자동 산출하여 적재합니다.
* **산출식**:
  * `FICTITIOUS_VAT_AMT = ROUND((PURCH_AMT * 6) / 106)`
  * 해당 컬럼은 상세 테이블 `OBSLPDTB.FICTITIOUS_VAT_AMT`와 헤더 테이블 `OBSLPHTB.FICTITIOUS_VAT_AMT`에 합산 집계되어 세무 감사용 기초 데이터로 활용됩니다.

### 4.2 헤더-상세 합산 집계 동기화 (`updateGoodsHd` / `updateOrdPurchHd`)
상세 테이블(`OBSLPDTB`)에 상품 수량이 수정되거나 신규 추가될 때마다 SQL Mapper 레벨에서 실시간으로 해당 전표에 귀속되는 전체 상세 행의 금액을 합산하여 헤더(`OBSLPHTB`)의 `PURCH_AMT`, `PURCH_VAT`, `FICTITIOUS_VAT_AMT`를 일괄 갱신(Sync)합니다.

## 5. 실시간 재고 및 회계 연쇄 작용 (Depth 3 Trigger & Procedure Flow)

매장 무발주 입고 확정(`PROC_FG = '4'`) 및 취소(`PROC_FG = '5'`) 트랜잭션이 완료되면 데이터베이스 내에서 아래와 같은 3단계 연쇄 작용이 실시간으로 수행됩니다.

### 5.1 [1단계] Java Trigger 발동 (`Tr_OBSLPD_T01_Service`)
상세 테이블(`OBSLPDTB`)의 진행 상태가 갱신되면 스프링 AOP/트리거 서비스를 통해 `processTrigger` 메서드가 가동됩니다.
1. **취소일 사전 정합성 검증**: 진행 상태가 `'5'`(취소)인 경우, 취소 기준 연월이 매입 연월과 일치하는지 검증합니다. (다를 경우 예외를 발생시켜 DB 롤백 처리)
2. **환산 수량 연산**: 수량에 낱개 입수수량을 곱해 최종 수불 수량을 산정합니다. (`qty = purchInQty * invInQty`)
3. **대체 기장 분개 분기**: 상품 마스터(`TGOODSTB`)의 세트 상품 여부(`SET_FG`)에 따라 호출할 실시간 재고 반영 코드베이스 수불 서비스를 판별합니다.

### 5.2 [2단계] Java 수불 서비스 실행 (`Sp_SUB_IMTRLG_I_Service`)
일반 상품의 경우 코드베이스로 전환된 `Sp_SUB_IMTRLG_I_Service` 가동을 통해 재고 원장에 영수증 및 전표 단위의 수불 데이터를 적재합니다.
* **수불 원장 기록 (`IMTRLGTB` 인서트)**:
  * 입고 확정 시: 매입 전표 정보와 수량(`qty`), 매입가액(`cost`)이 양수(+)로 신규 적재됩니다.
  * 입고 취소 시: 전표 번호를 연계 키(`KEY_BILL_NO`)로 하여 수량 및 금액이 마이너스(-) 음수로 적재됩니다.
* **수불 대기 정보 반영 (`IMTRLGTB` 적재)**:
  * 적재된 수불 정보는 `hmsfns.IMTRLGTB` 테이블에 `PROC_YN = 'N'` 상태로 기록되어 배치 처리를 대기합니다.
  * 실제 현재고 원장 테이블(`IMCRIOTB`)의 현재고(`CUR_QTY`) 및 일/월수불 원장(`IMDDIOTB`/`IMMMIOTB`)은 야간 또는 주기적으로 구동되는 재고 반영 배치 프로그램 (`DmIMTR01Service`)에 의해 최종 합산 가감산(`CUR_QTY = CUR_QTY + qty`)됩니다.

### 5.3 [3단계] 본사/매장 매입 이력 로그 적재 트리거 (`Tr_OBSLPH_T01_Service`)
헤더 테이블(`OBSLPHTB`)의 상태 갱신 완료 시점에 가동되어 매입 정산의 기초가 되는 `hmsfns.OBSLPLOG` 테이블에 로그 데이터를 인서트합니다.
* **로그 적재 내용**:
  * 확정 시: 해당 거래처 및 매장의 매입 공급가액 및 부가세를 양수(+)로 인서트합니다.
  * 취소 시: 정산 마감을 위해 취소 전표에 해당하는 역대체 금액을 음수(-) 마이너스로 분개 대체 적재하여 회계 대조 정합성을 100% 보장합니다.

### 5.4 재고 반영 확인 관련 화면 및 반영 시점
입고 확정/취소 처리한 재고 변동량은 배치 프로그램에 의해 최종 반영된 이후 화면을 통해 확인할 수 있습니다.
* **관련 확인 화면**: **[ST] 재고관리 > 재고현황 > 현재고조회 (`st_stock_00007`)**
* **동작 원리**: 
  * 사용자가 `st_vendor_00006` 화면에서 확정/취소 처리 시 `IMTRLGTB` 수불 대기 로그로 실시간 적재됩니다.
  * 이후 재고 반영 배치 프로그램 (`DmIMTR01Service`)이 실행되어 `IMTRLGTB` 로그를 읽고 일수불 (`IMDDIOTB`) 및 월수불 (`IMMMIOTB`) 데이터를 합산 갱신합니다.
  * 현재고조회 (`st_stock_00007`) 화면은 월수불 (`IMMMIOTB`)과 일수불 (`IMDDIOTB`) 수량의 합산값 (`MM_QTY + DD_QTY`)을 기준으로 현재고를 계산하여 출력하므로, **배치가 정상 구동되어 반영을 마친 시점부터 변경된 재고량이 화면에 즉시 조회**됩니다.

### 5.5 새벽배치(Batch) 연동 제외 및 비즈니스 이유
본 화면에서 등록 및 취소하는 무발주 전표(`ORDER_FG = '1'`)는 새벽배치 전송 대상에서 제외되도록 프로그래밍되어 있습니다.
* **배치 쿼리 및 프로시저 필터링 조건 (`ORDER_FG = '0'`)**:
  * 새벽 배치 프로그램(`ObslInerface_SqlMapper.xml`의 `selectObslTargetList` 쿼리) 및 EDB 새벽 송신용 프로시저(`SUB_IF_HYDM_SEND_P`의 `CURSOR OBSL` 커서 정의)에서 연동 대상 수집 시 **`AND A.ORDER_FG = '0'`** 조건이 명시되어 있습니다.
  * 따라서 무발주 입고(등록 시 `ORDER_FG = '1'`로 저장)는 새벽배치 대상에 포함되지 않습니다.
* **비즈니스적 사유**:
  * **발주 입고 (`ORDER_FG = '0'`)**: 본사 주도의 정식 발주 거래이므로 본사 회계 시스템(ERP)으로 전송하여 대금 세금 계산 및 통합 정산 지급을 새벽배치를 통해 원격 처리해야 합니다.
  * **무발주 입고 (`ORDER_FG = '1'`)**: 본사 주문 통제를 거치지 않는 매장 현장의 소액/긴급 거래이므로 본사 대금 정산 인터페이스로 전송되지 않으며, 매장 시재(현금)나 자체 현장 비용을 통해 로컬 정산 처리되기 때문에 배치가 불필요합니다.
