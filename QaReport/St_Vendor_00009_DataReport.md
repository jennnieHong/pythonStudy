본 보고서는 **매입발주현황 (st_vendor_00009)** 화면의 목록 검색 및 상세 조회(팝업) 시, DB 데이터의 상태와 조회 조건에 따라 화면에 어떤 전표가 노출되고 상세 팝업에서 **발주 수량(및 금액)**, **입고 수량(및 금액)**, **반품 수량(및 금액)** 필드가 어떻게 매핑되어 노출되는지에 대한 정밀 분석 보고서입니다.

## 1. 마스터 조회 목록 (`/search`) 데이터 노출 조건 및 케이스

### 1.1 진행구분별 파라미터 매핑 및 SQL 조건

## 2. 상세 내역 팝업 (`/detailSearch`)의 세 필드 노출 조건 및 결론

상세 조회 팝업에서는 선택된 전표의 세부 상품 품목별로 `발주수량/금액`, `입고수량/금액`, `반품수량/금액`이 표시됩니다. 이 데이터들은 전표의 **구분(`SLIP_FG`)**과 **진행상태(`PROC_FG`)**에 따라 값이 동적으로 렌더링됩니다.

### 2.1 결론: **단일 상품행(Row) 기준으로 세 필드가 동시에 실값(> 0)을 가질 수는 없습니다.**
화면 그리드 상에는 `발주`, `입고`, `반품` 컬럼이 나란히 존재하지만, 특정 전표를 클릭해 팝업을 열었을 때 특정 상품행에 세 항목 모두 동시에 `0`이 아닌 실제 값이 노출되는 케이스는 존재하지 않습니다.

### 2.2 SQL Mapper 데이터 매핑 조건 (`DECODE` 분석)
상세 데이터를 가져오는 `getDetailList` 쿼리의 컬럼 계산 식은 다음과 같습니다:
* **발주 수량**: `DECODE(OH.SLIP_FG, '0', OD.ORDER_QTY, 0)`
* **입고 수량**: `DECODE(OH.PROC_FG, 4, DECODE(OH.SLIP_FG, '0', OD.PURCH_QTY, 0))`
* **반품 수량**: `DECODE(OH.PROC_FG, 4, DECODE(OH.SLIP_FG, '1', OD.PURCH_QTY, 0))`

위 조건에 따른 데이터 노출 결과는 아래 4가지 조합 케이스로 분류됩니다.

## 3. 진행 상태 및 전표 조건에 따른 4가지 데이터 매핑 케이스

### [Case 1] 주문 전표 (`SLIP_FG = '0'`) & 입고확정 전 (`PROC_FG < '4'`)
* **노출 상태**: **발주** 필드만 활성화되어 실값이 표시되고, **입고** 및 **반품** 필드는 `0`으로 표시됩니다.
* **SQL 매핑 결과**:
  - 발주: `OD.ORDER_QTY` (실값)
  - 입고: `0` (`PROC_FG != 4` 이므로)
  - 반품: `0` (`SLIP_FG != '1'` 이므로)
* **화면 렌더링**: `[발주수량: N, 입고수량: 0, 반품수량: 0]`

### [Case 2] 주문 전표 (`SLIP_FG = '0'`) & 입고확정 완료 (`PROC_FG = '4'`)
* **노출 상태**: **발주** 필드와 **입고** 필드에 모두 실값이 노출되며, **반품** 필드는 `0`으로 표시됩니다.
* **SQL 매핑 결과**:
  - 발주: `OD.ORDER_QTY` (실값)
  - 입고: `OD.PURCH_QTY` (실값)
  - 반품: `0` (`SLIP_FG != '1'` 이므로)
* **화면 렌더링**: `[발주수량: N, 입고수량: M, 반품수량: 0]`
* **수치 불일치 가능성**: 발주 수량(`N`)과 입고 수량(`M`)은 실무 유통 흐름에 따라 **서로 다를 수 있습니다.** (상세 시나리오는 4절 참고)

### [Case 3] 반품 전표 (`SLIP_FG = '1'`) & 반품확정 완료 (`PROC_FG = '4'`)
* **노출 상태**: **반품** 필드에만 실값이 노출되고, **발주** 및 **입고** 필드는 `0`으로 표시됩니다.
* **SQL 매핑 결과**:
  - 발주: `0` (`SLIP_FG != '0'` 이므로)
  - 입고: `0` (`SLIP_FG != '0'` 이므로)
  - 반품: `OD.PURCH_QTY` (실값 - 반품된 수량)
* **화면 렌더링**: `[발주수량: 0, 입고수량: 0, 반품수량: K]`

### [Case 4] 반품 전표 (`SLIP_FG = '1'`) & 반품등록/미확정 (`PROC_FG < '4'`)
* **노출 상태**: 모든 필드가 `0`으로 표시됩니다. (미확정 반품은 수량 확정이 되지 않아 입출 실적으로 잡히지 않음)
* **SQL 매핑 결과**:
  - 발주: `0` (`SLIP_FG != '0'` 이므로)
  - 입고: `0` (`SLIP_FG != '0'` 이므로)
  - 반품: `0` (`PROC_FG != 4` 이므로)
* **화면 렌더링**: `[발주수량: 0, 입고수량: 0, 반품수량: 0]`

## 4. 입고 완료 단계 (Case 2)에서 발주와 입고 수치가 다를 가능성

주문 전표가 최종적으로 **입고확정(PROC_FG = '4')** 단계에 도달했을 때, 동일 상품에 대하여 `발주수량(ORDER_QTY)`과 `입고수량(PURCH_QTY)`이 일치하지 않는 실무 케이스는 매우 흔하게 발생하며 이는 정상적인 비즈니스 프로세스입니다.

### 4.1 불일치 발생 실무 시나리오
1. **부분 입고 및 배송 누락 (Under-delivery)**
   - 가맹점에서 거래처에 특정 상품 10개를 발주(`ORDER_QTY = 10`)했으나, 거래처 재고 부족 또는 배송 차량 적재 오류로 인해 8개만 매장에 배송된 경우. 매장 매니저는 실물 확인 후 8개만 검수하여 확정하므로 `PURCH_QTY = 8`이 됩니다.
2. **거래처 과배송 및 서비스 품목 가산 (Over-delivery)**
   - 거래처에서 임의로 더 많은 수량을 배송했거나(예: 10개 주문에 12개 배송), 유통기한이 임박한 원부자재를 증정 서비스 개념으로 가산하여 납품한 경우. 현장 검수 시 실재고 12개를 모두 정상 입고로 잡기로 합의하면 `ORDER_QTY = 10`, `PURCH_QTY = 12`가 됩니다.
3. **입고 검수 시 현장 반품 및 수량 제외 (Inspection Rejection)**
   - 발주된 10개가 배송되었으나, 검수 과정에서 2개가 파손되거나 유통기한이 지난 것이 확인되어 그 자리에서 납품 기사에게 즉시 반환(현장 조정)하고 최종 8개만 입고 완료 서명한 경우, `ORDER_QTY = 10`, `PURCH_QTY = 8`로 저장됩니다.

## 5. 데이터베이스 실제 데이터 검증 결과

실제 DB(`OBSLPHTB`, `OBSLPDTB`, `MGOODSTB`) 상에서 조회 조건 및 데이터 성격을 검증한 결과, 아래와 같이 데이터가 적재되어 있음을 확인할 수 있습니다.

### 5.1 검증 쿼리
```sql
SELECT D.MS_NO        AS "매장코드"
     , D.ORDER_DATE   AS "발주일자"
     , D.SLIP_NO      AS "전표번호"
     , D.GOODS_CD     AS "상품코드"
     , G.GOODS_NM     AS "상품명"
     , D.ORDER_QTY    AS "발주수량"
     , D.PURCH_QTY    AS "입고수량"
     , H.PROC_FG      AS "진행상태"
     , H.SLIP_FG      AS "전표구분"
  FROM hmsfns.OBSLPDTB D
  JOIN hmsfns.OBSLPHTB H 
    ON D.ORDER_DATE = H.ORDER_DATE 
   AND D.MS_NO      = H.MS_NO 
   AND D.SLIP_NO    = H.SLIP_NO 
   AND D.SLIP_FG    = H.SLIP_FG
  JOIN hmsfns.MGOODSTB G
    ON D.GOODS_CD   = G.GOODS_CD
   AND D.MS_NO      = G.MS_NO
 WHERE D.MS_NO = 'NC0007'
 ORDER BY D.ORDER_DATE, D.SLIP_NO, D.GOODS_CD;
```

## 6. 요약 및 시사점

1. **사용자 UI 일관성**:
   - 화면 설계상 발주, 입고, 반품 수량 필드가 고정 컬럼으로 존재하지만, 전표 성격(`SLIP_FG`)에 따라 특정 필드들은 0으로 표시되는 것이 정상입니다.
   - 발주 수량과 입고 수량의 차이는 정상적인 유통 실무 데이터의 변동 사항을 그대로 반영한 것이므로 데이터 정합성에 이상이 없습니다.
2. **백엔드 무결성**:
   - `ORDER_QTY`와 `PURCH_QTY`가 독립된 컬럼으로 관리되며 이력의 변동을 온전히 보존하고 있으므로 데이터 정합성에 이상이 없습니다.
