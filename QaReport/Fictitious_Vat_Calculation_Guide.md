# Fictitious VAT (의제매입세액) 데이터 처리 및 자동계산 가이드

본 가이드는 영업정보시스템(F&B) 내 면세 식자재 매입 시 발생하는 **의제매입세액(Fictitious VAT)의 데이터 연산 로직, DB 스키마 구조, 금액 집계 메커니즘 및 연계 화면 목록**을 상세히 정리한 문서입니다.

---

## 1. 개요 및 비즈니스 배경
* **의제매입세액 공제**: 부가가치세법상 면세로 구입한 농산물, 축산물, 수산물, 임산물 등을 원재료로 하여 제조/가공한 재화에 대해 일정율(예: 6/106)의 세액을 매입세액으로 간주하여 공제해주는 제도입니다.
* **F&B 시스템 적용**: 매장에서 양파, 고기 등의 면세 식자재를 매입할 때 발생하며, 매입 확정 시 시스템이 상품 마스터 정보를 기반으로 의제세액을 자동으로 계산하여 전표 상세 및 헤더 테이블에 기록합니다.

---

## 2. 의제매입세액의 세무적 대전제 (면세 대상 상품 필수)

의제매입은 **법적으로 반드시 "면세(Tax-Free)"로 구입한 농·수·축·임산물 원재료에 대해서만 적용**할 수 있습니다. 

* **과세 상품과의 차이**:
  * **과세 상품 (일반 식재/공산품 등)**: 매입할 때 이미 실제 부가세 10%를 지불하고 매입하므로, 세금계산서를 통해 10%의 세액을 그대로 일반 매입세액공제로 환급받습니다. (의제매입을 적용할 필요도 없고 불가능함)
  * **면세 상품 (순수 농수산물 등)**: 매입 시 부가세가 면제되어 거래(계산서 발행)되므로 공제받을 매입세액이 원래는 존재하지 않습니다. 그러나 이를 원재료로 하여 과세 재화(음식 등)를 가공/판매하는 사업자의 세금 부담을 경감하기 위해, 부가세가 포함되어 있던 것으로 **의제(간주)**하여 면세 매입금액의 일부를 세무서에서 환급받도록 해 줍니다.

## 4. 프로그램 내 `TAX_FG` 임시 판정 코드 ('2' 판정 기준)

시스템의 품의서 작성 및 매입 처리 로직에서는 상품 마스터(`TGOODSTB`)의 원본 `TAX_FG`와 `FICTITIOUS_YN` 플래그를 조합하여 내부 계산용 **임시 `TAX_FG`**를 아래 3가지 기준으로 새로 판정합니다.

### 4.1 판정 기준 매핑 테이블
| 원본 `TAX_FG` (과면세) | 원본 `FICTITIOUS_YN` (의제여부) | 변환 후 `TAX_FG` (계산용) | 판정 분류 및 비즈니스 의미 |
|:---:|:---:|:---:|:---|
| **`'0'` (면세)** | **`'Y'` (의제)** | **`'2'`** | **계산서 (의제매입세액 공제 대상)** |
| `'1'` (과세) | `'N'` (비의제) | `'1'` | 과세 (포함) 품목 |
| 그 외 모든 조합 | - | `'0'` | 일반 면세 품목 (의제세액 공제 불가) |

> [!IMPORTANT]
> - 이 조건에 따라 과세 상품(`TAX_FG = '1'`)은 의제여부가 `'Y'`이더라도 의제매입(2)으로 분류되지 않고 무조건 일반 과세(1)로 처리되어 세법적 정합성이 강제됩니다.

### 4.2 SQL 판정 쿼리 예시 (`Hq_Vendor_00002_Sql.xml`)
```sql
CASE WHEN TG.TAX_FG = '0' AND TG.FICTITIOUS_YN = 'Y' THEN '2' -- 의제매입 판정 (면세 + 의제대상만 해당)
     WHEN TG.TAX_FG = '1' AND TG.FICTITIOUS_YN = 'N' THEN '1' -- 일반과세 판정
     ELSE '0'                                                 -- 일반면세 판정
 END AS TAX_FG
```

## 5. 자동 계산 연산 공식 및 로직

매입 및 입고가 확정되어 매입 데이터가 생성되는 시점(예: 발주품의서 확정 시)에 위의 **`TAX_FG = '2'` (의제매입 대상)**로 판정된 품목들에 한해 아래의 공식에 따라 프로그램 단에서 자동 계산을 수행합니다.

### 5.1 계산 공식
* **단가 기준 의제부가세액 (`ficTax`)**:
  $$\text{ficTax} = \text{Math.round}\left(\text{매입단가(unitPrice)} \times \frac{6}{106}\right)$$
* **총 의제부가세액 (`ficTaxAmt`)**:
  $$\text{ficTaxAmt} = \text{Math.round}\left(\text{매입금액(orderAmt)} \times \frac{6}{106}\right)$$

> [!NOTE]
> - 계산에 적용되는 요율인 **`6/106`**은 시스템 내부 프로그램 및 쿼리상에 고정 반영되어 연산됩니다.
> - 산출된 세액은 부가세 계산 로직과 마찬가지로 소수점 첫째 자리에서 반올림(`Math.round`)하여 원 단위로 적재됩니다.

### 5.2 Java 서비스 소스 예시 (`Hq_Vendor_00002_Service.java`)
```java
// 의제매입 상품 판정 (TAX_FG = '2')
if (requestDetailList.get(j).getTaxFg().equals("2")) {
    ficTax    = Math.round(unitPrice * 6 / 106);
    ficTaxAmt = Math.round(orderAmt * 6 / 106);
}
```

## 6. 금액 표출 및 집계 메커니즘 (전체와 부분 금액)

의제매입세액이 포함된 전표의 금액 집계 및 화면 표출은 임의의 비율로 쪼개는 것이 아니라, **실제 거래된 품목별 실거래가 기반의 개별 집계(전체 및 부분 금액 표시)** 방식을 따릅니다.

### 6.1 데이터 성격
* **매입/입고 기준**: 매출(판매) 가격이 아니라 본사가 거래처로부터 원재료를 사들인 **매입/입고(Inbound/Purchase)** 데이터를 기준으로 합니다.

### 6.2 금액 분리 및 합산 규칙
하나의 전표 안에 과세 상품과 의제매입 면세 상품이 섞여 있는 경우, 상세 테이블(`OBSLPDTB`)의 품목별 데이터가 다음과 같이 개별 영역으로 분류되어 누적 집계됩니다.

* **과세 영역 (부분 금액)**: 일반 과세 품목의 공급가 및 부가세 합산
* **의제매입 영역 (부분 금액)**: 의제매입 품목의 순수 공급가(매입가 - 의제세액) 및 의제세액 합산
* **당월 합계 (전체 금액)**: 과세 합계와 의제 합계를 모두 더한 **전체 실거래 총 매입액**

### 6.3 금액 분리 집계 예시 (총 매입액 15,000원 입고 시)
* **품목 A (과세 상품)**: 매입가 10,000원, 부가세 1,000원
* **품목 B (의제매입 면세 상품)**: 매입가 4,000원 (의제세액 226원 자동 계산)

| 집계 영역 | 구분 | 금액 | 비고 |
|-----------|------|------|------|
| **과세 영역 (부분)** | 공급가 | `10,000원` | 품목 A의 공급가 |
| | 부가세 | `1,000원` | 품목 A의 부가세 |
| | 합계 | `11,000원` | 품목 A의 총액 |
| **의제매입 영역 (부분)**| 공급가 | `3,774원` | 품목 B의 순수 공급가 (4,000 - 226) |
| | 의제세액 | `226원` | 품목 B의 공제세액 |
| | 합계 | `4,000원` | 품목 B의 총액 |
| **당월 합계 (전체)** | **총 매입 합계** | **`15,000원`** | **과세 합계(11,000) + 의제 합계(4,000)** |

## 8. 최근 식별된 공제 정산 결함 (QA 피드백)

의제매입 공제현황 E2E 및 소스 코드 검증 중 발견된 주요 결함 사항입니다.

### 8.1 가맹점 화면 반품 전표 합계 부호 오류 (SQL 결함)
* **대상 파일**: [St_Vendor_00014_Sql.xml](file:///d:/workspace/hmotors/workspace_hms20260326/backoffice/hyundai-backoffice-webapp/src/main/resources/sqlmapper/vendor/St_Vendor_00014_Sql.xml) L21
* **현상**: 반품 전표(`SLIP_FG = '1'`)의 합계(`PURCH_SUM`) 계산 시, 수량과 세액의 합에 부호 반전(`*-1`)이 올바르게 적용되지 않아 플러스(+) 값으로 출력됨.
* **원인**: `B.PURCH_AMT + NVL(B.FICTITIOUS_VAT_AMT,0)*-1` 과 같이 작성되어 연산자 우선순위에 의해 의제세액만 마이너스로 곱해지고 공급가액은 플러스로 유지됨.
* **조치**: `(B.PURCH_AMT + NVL(B.FICTITIOUS_VAT_AMT,0)) * -1` 형태로 괄호를 보강해야 함.

### 8.2 가맹점 테이블 마이너스 수량/금액 미표출 오류 (JS 결함)
* **대상 파일**: [st_vendor_00014_bt.js](file:///d:/workspace/hmotors/workspace_hms20260326/backoffice/hyundai-backoffice-webapp/src/main/webapp/WEB-INF/views/backoffice/main/contents/st/vendor/st_vendor_00014/js/st_vendor_00014_bt.js) L214
* **현상**: 반품 전표의 마이너스(-) 수량 및 공급가액이 테이블 상에서 `0`으로 대치되어 표출됨.
* **원인**: `numberFormatter` 내부에 `if(parseInt(value) > 0)` 필터가 존재하여, 0 이하인 음수 값이 유입될 경우 무조건 `0`을 반환함.
* **조치**: 본사 화면(`hq_vendor_00016_bt.js`)의 포맷터처럼 음수 값도 정상적으로 출력할 수 있도록 필터 조건을 제거해야 함.

### 8.3 소수점 단위(1 미만 실수) 수량 표출 오류 (JS 결함)
* **대상 파일**: [st_vendor_00014_bt.js](file:///d:/workspace/hmotors/workspace_hms20260326/backoffice/hyundai-backoffice-webapp/src/main/webapp/WEB-INF/views/backoffice/main/contents/st/vendor/st_vendor_00014/js/st_vendor_00014_bt.js) L214
* **현상**: 단위가 Kg인 품목 등 입고 수량이 소수점 단위(예: `0.1` Kg 등 1 미만의 실수)일 경우 화면 상에 무조건 `0`으로 오표기되는 현상 식별.
* **원인**: `numberFormatter` 내부에 정수형 변환 조건인 `if(parseInt(value) > 0)`이 설계되어, `parseInt("0.1")` 수행 결과가 `0`이 되어 0 이하 분기(`else`)를 타고 무조건 `0`을 반환하게 됩니다.
* **조치**: 정수 절사를 유발하는 `parseInt` 대신 실수형 변환을 지원하는 `parseFloat`를 사용하거나, 단순히 값이 존재하는지 체크(`value != null && value !== ''`)하도록 수정해야 합니다.
