# Hq_System_00001 화면 마이그레이션 및 테스트 종합 보고서

본 보고서는 `Hq_System_00001` (POS 버전 등록 및 매장 배포 관리) 화면에 대한 Oracle -> EPAS(PostgreSQL) 마이그레이션 에러 해결 과정, 자동화 테스트 중 발생한 한계점, 할루시네이션(시행착오) 및 향후 과제를 종합적으로 요약한 문서입니다.

---

## 1. 작업 수행 내역 (과정)

1. **에러 식별 및 원인 파악**
   - 파일 업로드 테스트 중 `500 Server Error` 발생.
   - 로그 분석 결과 `operator does not exist: text + integer` 에러 식별. (Oracle의 묵시적 형변환이 EPAS의 엄격한 타입 검사에 의해 차단됨을 확인)
2. **SQL 문법 수정 (`Hq_System_00001_Sql.xml`)**
   - 채번 쿼리(`getVersionSeq`) 등에서 `VER_SEQ` 필드를 `INTEGER`로 명시적 `CAST` 하도록 수정하여 덧셈 연산 호환성 확보.
3. **공통 모듈 연쇄 에러 해결 (`CommonModule_GoodsClass_Sql.xml`)**
   - `getFileInfo` 쿼리 내 난수 생성 로직에서 연쇄 에러(`function mod(double precision, integer) does not exist`, `function chr(numeric) does not exist`) 확인.
   - `MOD` 함수 내부의 `FLOOR` 연산 결과를 `INTEGER`로 이중 명시적 캐스팅(`CAST(FLOOR(...) AS INTEGER)`)하여 모든 문법 에러 해결 완료.
4. **DB 정합성 검증 완료**
   - Python 스크립트를 통해 개발 EPAS DB(`192.168.10.206/edb`)에 접속하여 데이터 검증 수행.
   - `hmsfns.MVERSNTB`(버전 마스터) 및 `hmsfns.MVERSMTB`(매장 배포 매핑) 테이블에 UI 조작 내역(등록, 배포확정, 배포취소 상태 변경)이 완벽히 Commit 되었음을 확인.

---

## 2. 직면한 어려움 및 자동화 테스트의 한계 (Limitations & Difficulties)

1. **OS 레벨 팝업 제어의 한계 (프로세스 마비 현상)**
   - 브라우저 자동화 AI 에이전트가 `<input type="file">`을 클릭하여 윈도우 OS의 기본 '파일 탐색기' 창이 열렸을 때, 브라우저 DOM을 벗어난 OS 레벨의 창을 제어하지 못했습니다.
   - 이로 인해 에이전트의 프로세스가 멈추고 대기 상태에 빠지는(Freezing) 치명적인 한계가 발생했습니다.
2. **좌표 인식 깨짐 및 세션 단절**
   - OS 창 제어 실패의 여파로 에이전트와 브라우저 간의 디버깅 프로토콜(DevTools Protocol) 통신이 끊어지거나(`EOF`), 화면 Viewport를 `0x0`으로 오인식하여 이후 물리적 클릭 명령이 튕겨 나가는 시스템 마비 현상을 겪었습니다.
3. **복잡한 그리드 UI에 대한 시각 인지 오류 (무한 루프)**
   - 강제 재기동 후 에이전트가 매장 선택 및 배포 확정/취소를 수행하려 했으나, 2단 분리된 그리드 화면과 다수의 체크박스 상태, 알림 팝업들을 완벽히 인지하지 못했습니다.
   - 이로 인해 스스로 동작 성공 여부를 확신하지 못하고, 동일한 [저장] 및 [배포확정] 버튼을 수십 번 무한 반복 클릭하는 루프(Loop) 현상에 빠져 사용자가 개입하여 강제 취소해야만 했습니다.

## 3. 할루시네이션(Hallucination) 및 시행착오

1. **EPAS 호환성 기능에 대한 오판 및 과도한 수정**
   - **문제**: EPAS가 Oracle 호환성 모드를 강력히 지원하여 `NVL`, `SYSDATE` 등의 함수를 기본적으로 사용할 수 있음에도 불구하고, 이를 알지 못해 PostgreSQL 표준 문법인 `COALESCE`나 `NOW()`로 불필요하게 일괄 교체하려 했던 판단 착오(할루시네이션)가 있었습니다.
2. **Alias 누락에 의한 `NullPointerException` 리스크 (사용자 지적으로 정립)**
   - **문제**: Oracle과 달리 EPAS 환경에서 `NVL`을 사용할 경우 Alias 구문(`AS 컬럼명`)이 없으면 리턴되는 컬럼명이 소문자 `nvl`로 치환됩니다.
   - **시행착오**: 기존 코드를 무심코 사용할 경우, MyBatis와 Java `HashMap` 단에서 `map.get("컬럼명")`을 찾을 수 없어 500 에러(NPE)가 터지는 치명적인 함정이 있었으나, 이를 사전에 인지하지 못하고 사용자님의 예리한 지적을 받고 나서야 그 심각성을 정립하게 되었습니다.

## 4. 추가적으로 필요한 작업 (Next Steps)

1. **전사 SQL 매퍼 파일 Alias 전수 검사 (매우 시급)**
   - `NVL`, `COALESCE`, `MAX` 등의 함수를 사용하는 전체 쿼리 파일에 대해, `AS 컬럼명` 이 누락된 쿼리가 없는지 정규식 스크립트를 통한 전수 조사가 필요합니다. 
   - (조치 방안: Java 단의 NullPointerException 잠재 위험 제거)
2. **공통 모듈 사이드 이펙트 모니터링**
   - 금번 수정된 `CommonModule_GoodsClass_Sql.xml`의 `getFileInfo` (채번 및 난수 조합) 쿼리는 타 화면에서도 공통으로 호출되므로, 다른 모듈의 첨부파일 등록 기능에서도 정상적으로 채번이 이루어지는지 회귀 테스트가 요구됩니다.
3. **파일 업로드 기능에 대한 E2E 자동화 우회 파이프라인 구축**
   - OS 파일 탐색기가 팝업되는 브라우저 테스트 방식은 자동화 한계가 명확하므로, 향후에는 UI 클릭이 아닌 파이썬 스크립트 기반의 **Multipart API Request를 백엔드에 직접 쏘는 방식**의 API E2E 테스트 시나리오로 전환하여 테스트 커버리지를 확보해야 합니다.
