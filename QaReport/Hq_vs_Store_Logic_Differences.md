# [분석 리포트] 본부(HQ)와 매장(Store)의 비즈니스 로직 및 시스템 처리 차이 분석
**작성일**: 2026-06-09  
**작성자**: AI QA Agent (Antigravity)  
**대상 시스템**: 현대 모터스튜디오 영업정보시스템 (HMS Backoffice)

---

## 1. `chain_hq_yn` 개요 및 중요성
`chain_hq_yn` 컬럼은 매장 기본 정보 테이블(`hmsfns.mmenbstb`)에 정의되어 있으며, 해당 매장이 **체인 본사(Headquarters, 'Y')**인지 **일반 매장(Store/Branch, 'N')**인지를 식별하는 핵심 플래그입니다.
이 플래그는 사용자의 권한 필터링, 데이터 조회 스코프 제어, 비즈니스 트랜잭션 분기에 핵심적인 역할을 수행합니다.

---

## 2. `chain_hq_yn` 검증/체크 위치 및 역할

프로그램 전체 소스코드 분석 결과, `chain_hq_yn`은 주로 MyBatis SQL Mapper의 XML 쿼리문 조건절에서 다음과 같이 체크되어 데이터를 제어하고 있습니다.

### 2.1 사용자 로그인 및 권한 스코프 결정 (`UserAuth_Sql.xml`)
* **역할**: 로그인 성공 시 세션을 구성할 때 사용자의 소속 매장(`muserstb.ms_no`)을 기준으로 본사 매장 여부를 판별합니다.
* **주요 쿼리**:
  ```xml
  /* UserAuth_Mapper - selectUserAuth */
  SELECT X.CHAIN_HQ_YN
    FROM hmsfns.mmenbstb X
   WHERE X.ms_no = A.ms_no
  ```
* **결과**: `chain_hq_yn = 'Y'`인 사용자는 본부 관리자(또는 본사 직원) 세션을 획득하여 마스터 제어 및 전체 매출/재고 통계 화면에 접근할 수 있게 되며, `'N'`인 사용자는 자신이 속한 단일 매장의 로컬 정보만 제어하도록 차단됩니다.

### 2.2 마스터 데이터 및 통계 집계에서의 분기
* **실제 매장 수 집계 (`Admin_Master_00003_Sql.xml`)**:
  * 체인 마스터 조회 시 산하의 "영업 매장 개수"를 집계할 때 본사 매장을 카운트에서 제외하기 위해 `'N'` 조건을 사용합니다.
  ```xml
  SELECT COUNT(*) FROM hmsfns.mmenbstb M WHERE M.chain_no = A.chain_no AND M.chain_hq_yn = 'N'
  ```
* **매출 및 실적 통계 (`Hq_Sales_00001_Sql.xml`, `Hq_Sales_00002_Sql.xml`)**:
  * 본사 계정에서 직접 등록한 테스트 전표나 본사 내부 실적은 실제 매장 매출 합산 시 왜곡을 방지하기 위해 제외합니다.
  ```xml
  AND B.chain_hq_yn = 'N' -- 통계 대상에서 본사 데이터 배제
  ```

### 2.3 재고 및 점간이동 대상 매장 필터링 (`Hq_Stock_00009_Sql.xml`, `St_Stock_00005_Sql.xml` 등)
* **역할**: 점간이동 등록 시 "보내는 매장" 및 "받는 매장" 선택 콤보박스 로딩 시 본사 매장은 제외해야 하므로 `'N'`인 영업 매장 리스트만 쿼리하여 바인딩합니다.
* **주요 쿼리**:
  ```xml
  SELECT ms_no, ms_nm 
    FROM hmsfns.mmenbstb 
   WHERE chain_no = #{chainNo} 
     AND chain_hq_yn = 'N' -- 본사는 드롭다운에서 필터링
  ```

## 3. 본부(HQ)와 매장(Store)의 주요 로직 차이 비교

| 분류 | 본부 (HQ / `chain_hq_yn = 'Y'`) | 매장 (Store / `chain_hq_yn = 'N'`) |
|:---|:---|:---|
| **화면 접두사** | 주로 **`hq_`**로 시작 (예: `hq_stock_00011` 본사 점간이동) | 주로 **`st_`**로 시작 (예: `st_stock_00010` 매장 점간이동) |
| **데이터 조회 스코프**| 체인 전체(`C001`, `C002` 등) 내 **모든 매장**의 재고, 매출, 전표를 통합 제어 및 조회 가능 | **본인 소속 매장**(`ms_no`)의 로컬 데이터만 엄격하게 분리/제어 (타 매장 데이터 접근 차단) |
| **점간이동 처리 권한**| **본사 직권 강제 종결**: 본사 점간이동 화면에서 전표 확정 시, 보내는 매장 확정(`PROC_FG='1'`)과 받는 매장 확정(`PROC_FG='2'`)이 **한 번에 트랜잭션 처리**되어 즉시 수신 완료됩니다. | **상호 합의형 프로세스 (Handshake)**:<br>1. 출고처 매장 사용자가 출고 확정 (`PROC_FG = '1'`) 처리<br>2. 입고처 매장 사용자가 물량 검수 후 직접 입고 확정 (`PROC_FG = '2'`) 처리해야 트랜잭션 종결 |
| **매출 마감 및 원가 정산**| 각 매장에서 취합된 매출/수불 내역을 바탕으로 체인 단위의 **배치 집계 및 총평균단가 마감**을 연동 실행하는 주체 | 일별 매출 마감, 수기 조정재고 등록 등 **원천 데이터(DML)를 생성**하고 제공하는 대상체 |
| **재고 조정 권한** | 본사가 직접 매장의 재고 수량을 강제 보정하거나 원가를 갱신할 수 있음 (조정 재고 전표 직접 처리 등) | 정해진 당월 마감 기한 내에만 본인 매장의 실사 재고 등록 및 조정 가능 |

## 4. 입고 및 반품 업무에서의 출고 공급가와 입고 공급가 산정 로직

입고 현황 관련 화면들(매장/본사 공통)에서 그리드 및 상세 팝업에 표출되는 **출고 공급가액**과 **입고 공급가액**은 비즈니스 처리 단계에 따라 다르게 산정되며, 특정 예외 상황에서는 두 금액 간에 차이가 발생할 수 있습니다.

### 4.1 산정 공식 (DB 테이블: `hmsfns.obslpdtb` 상세 내역 기준)
* **공급 단가 (단일 기준가)**: 디테일 테이블의 `PURCH_UCOST` 컬럼을 공통 기준으로 사용합니다.
* **출고 공급가액 (`SUPPLY_AMT`)**:
  * **정의**: 거래처(공급처)가 출고 시 전표에 등록한 금액입니다.
  * **공식**: `SUPPLY_QTY` (출고/공급 수량) $\times$ `PURCH_UCOST` (공급 단가)
* **입고 공급가액 (`PURCH_AMT`)**:
  * **정의**: 매장에서 최종 입고 검수 및 매입 확정한 금액입니다.
  * **공식**: `PURCH_QTY` (최종 매입 확정 수량) $\times$ `PURCH_UCOST` (공급 단가)

### 4.2 두 공급가액이 다르게 나타나는 원인 (물량 불일치 시나리오)
통상적으로 물류에 이상이 없고 검수가 정상 완료된 전표들은 **[출고(공급) 수량 = 실제 입고(매입) 수량]**이 되므로 두 공급가액이 동일하게 표출됩니다. 하지만 아래와 같은 유통 흐름상의 예외 발생 시 두 가액이 다르게 조회될 수 있습니다:

1. **배송 중 파손 또는 분실 (수량 차이)**:
   * 거래처는 10개를 출고(`SUPPLY_QTY = 10`)하였으나 배송 중 일부가 유실/파손되어, 매장에서는 실제 8개만 인수 및 입고 확정(`PURCH_QTY = 8`)한 경우.
   * **출고 공급가액** ($10\text{개} \times \text{단가}$) vs **입고 공급가액** ($8\text{개} \times \text{단가}$)
2. **검수 오차 및 불량품 반품(반입) 거부**:
   * 매장 검수 시 신선도 불량이나 하자가 있는 일부 수량에 대해 입고를 거부하고, 정상 상품만 확정 처리하여 `PURCH_QTY`가 `SUPPLY_QTY`보다 적게 등록된 경우.

## 5. 종합 평가
`chain_hq_yn` 플래그는 본사와 매장 간의 강력한 세션 및 데이터 격리를 처리하기 위한 기본 아키텍처로 사용됩니다. 
* **본사 모듈**은 모든 영업 매장(`chain_hq_yn = 'N'`)을 관제하고 데이터를 관리/정산하는 통제형 로직으로 설계되어 있습니다.
* **매장 모듈**은 실질적인 영업 및 재고 실사 전표를 생성하고, 개별 매장의 고유 업무만을 격리하여 수행하도록 보안과 책임 범위를 한정하는 분산형 로직으로 일관되게 구현되어 있습니다.
