# 발주품의작성 (Hq_Vendor_00002) 업무 흐름 및 데이터 가이드

본 문서는 가맹점의 구매의뢰 등록부터 본사의 검토, 확정, 품의 승인, 발주 및 최종 입고까지 이어지는 **표준 업무 흐름(Happy Path)**을 기준으로 데이터의 상태 변화를 이해하기 쉽게 정리한 가이드입니다.

---

## 1. 표준 업무 프로세스 흐름 (Happy Path)

가맹점(매장)이 등록한 임시 전표를 본사에서 직접 검토하고 확정한 뒤, 품의 승인 및 발주 처리를 거쳐 최종 입고에 이르는 표준적인 비즈니스 흐름입니다.

### 1.1 표준 흐름도 (End-to-End)

```text
[1단계: 매장] 구매의뢰 등록 및 저장
  └─ 화면: st_vendor_00001 (구매의뢰등록)
  └─ 데이터: OBREQHTB.proc_fg = '0' (의뢰등록/임시저장)
  └─ 특이사항: 매장 화면에서는 [확정] 버튼이 비활성화(주석 차단)되어 있어 오직 저장만 가능합니다.
        ▼
[2단계: 본사] 구매의뢰 검토 및 확정
  └─ 화면: hq_vendor_00001 (구매의뢰요청관리 - 본사)
  └─ 데이터: OBREQHTB.proc_fg = '0' ➔ '5' (의뢰확정)
  └─ 역할: 본사 관리자가 각 매장의 임시 저장 건('0')을 검토하여 최종 확정 처리합니다.
        ▼
[3단계: 본사] 발주 품의 승인
  └─ 화면: hq_vendor_00002 (발주품의작성 - 현재)
  └─ 데이터: OBREQHTB.proc_fg = '5' ➔ '1' (품의처리/완료)
        ▼
[4단계: 본사] 최종 발주 확정 (매입전표 생성)
  └─ 화면: hq_vendor_00002 (발주품의작성 - 현재)
  └─ 데이터: OBREQHTB.proc_fg = '1' ➔ '3' (발주확정)
            ※ 동시에 매입발주 테이블(OBSLPHTB)에 proc_fg = '1' (검수대기) 전표 신규 적재
        ▼
[5단계: 본사] 발주서 메일 전송
  └─ 화면: hq_vendor_00003 (발주전송관리)
  └─ 데이터: OBSLPHTB.purch_send_yn = 'S' (메일 이관 완료)
        ▼
[6단계: 매장] 상품 입고 검수 및 확정
  └─ 화면: st_vendor_00004 (매장입고관리)
  └─ 데이터: OBSLPHTB.proc_fg = '1' ➔ '4' (입고확정)
            ※ 이 단계에서 데이터베이스 트리거가 기동되어 매장의 실제 재고(CUR_QTY)가 최종 반영됨
```

## 2. 각 단계별 상세 데이터 처리 및 상태 전이

정상 흐름이 진행되는 과정에서의 구체적인 데이터 상태값(`proc_fg`)의 변화 메커니즘입니다.

### 2.1 [1단계] 구매의뢰 등록 (`st_vendor_00001`)
* **임시저장 시**: 매장에서 상품을 장바구니에 담고 [저장]하면 **`proc_fg = '0'` (의뢰등록)** 상태의 전표가 생성됩니다. 매장 화면에서는 본사로 직접 전송하는 기능이 제공되지 않으므로, 이 상태에서 작성 단계가 완료됩니다.
* **거래처 자동 지정 규칙**: 상품 등록 시 시스템이 상품을 공급하는 가장 적절한 거래처를 백엔드에서 자동 추적하여 **기본거래처(`purch_bas_vendor`)**와 **확정거래처(`purch_conf_vendor`)**에 동일한 값으로 세팅합니다. (상세 판단 기준은 FAQ Q5 참조)

### 2.2 [2단계] 구매의뢰 검토 및 확정 (`hq_vendor_00001`)
* **의뢰확정 시**: 본사 관리자가 매장에서 올린 임시 전표(`'0'`)들을 조회하여 내역을 확인한 뒤 **[확정]** 버튼을 클릭하면 **`proc_fg: '0' ➔ '5' (의뢰확정)`**으로 업데이트되어 본사 품의 대상 전표로 이관됩니다.

### 2.3 [3단계] 발주 품의 승인 (`hq_vendor_00002`)
* **품의처리 시**: 본사 관리자가 요청된 단가/수량/거래처를 검토하고 **[품의처리]**를 누르면 **`proc_fg: '5' ➔ '1' (품의처리/완료)`**로 변경되어 승인 상태가 됩니다.

### 2.4 [4단계] 최종 발주 확정 (`hq_vendor_00002`)
* **발주확정 시**: 본사 관리자가 승인된 전표를 선택하고 **[발주확정]**을 누르면 **`proc_fg: '1' ➔ '3' (발주확정)`**으로 전이됩니다.
* **매입전표 생성**: 이와 동시에 실질적인 협력업체 발주서 양식이 되는 매입발주 테이블(`OBSLPHTB`, `OBSLPDTB`)에 데이터가 새롭게 삽입(INSERT)되며, 이 생성된 매입전표의 최초 상태는 **`proc_fg = '1'` (검수대기/발주확정)**이 됩니다.
  *(※ 이 단계에서는 수불 및 재고 변동이 발생하지 않는 수불 배제 스펙이 구동됩니다.)*

### 2.5 [5단계] 발주서 메일 전송 (`hq_vendor_00003`)
* **메일전송 시**: 발주전송관리 화면에서 거래처 이메일을 확인하고 [메일전송]을 누르면, 외부 메일 서버로 이관되면서 전송 성공 플래그(`purch_send_yn = 'S'`)가 표시됩니다.

### 2.6 [6단계] 상품 입고 검수 및 확정 (`st_vendor_00004`)
* **입고확정 시**: 실물 상품이 배송되어 수량을 최종 확인하고 [입고확정]을 완료하면, 매입발주 테이블(`OBSLPHTB`)의 상태값이 **`proc_fg: '1' ➔ '4' (입고확정)`**으로 최종 업데이트됩니다.
* **최종 재고 가산**: 상태가 `'4'`로 갱신되는 순간, 백엔드 Java 트리거 서비스가 기동하여 가맹점의 상품 실재고(`MGOODSTB.CUR_QTY`)가 증가하고 수불 이력대장(`IMTRLGTB`) 및 로그 테이블(`obslplog`)에 변경 이력이 기록됩니다.

## 3. 취소 / 반려 프로세스 (Exception Path)

정상 흐름 도중 오류가 발생하여 데이터를 원상복구(취소)해야 할 때의 처리 메커니즘입니다.

### 3.1 품의 단계 취소 (품의취소)
* **대상 상태**: **`'1'` (품의처리/완료)** 상태일 때만 취소 가능합니다.
* **동작**: [품의취소] 버튼을 클릭하면 구매의뢰서 상태가 **`proc_fg: '1' ➔ '2' (품의취소/반려)`**로 변경됩니다.
* **이후 흐름**: 반려된 전표는 본사 관리자가 단가/수량 등을 수정한 후 다시 **[품의처리]**를 눌러 **`proc_fg: '2' ➔ '1'`**로 복원(승인)시킬 수 있습니다.

### 3.2 발주 단계 취소 (발주취소)
* **대상 상태**: **`'3'` (발주확정)** 상태일 때만 취소 가능합니다.
* **동작**: [발주취소] 버튼을 클릭하면 구매의뢰서 상태가 **`proc_fg: '3' ➔ '4' (발주취소)`**로 변경됩니다.
* **데이터 삭제**: 이와 동시에 발주확정 단계에서 자동 생성되었던 매입발주 전표(`OBSLPHTB`, `OBSLPDTB`) 데이터는 DB에서 물리적으로 **완전 삭제(DELETE)**됩니다.
* **이후 흐름**: 발주가 취소된 전표(`'4'`)는 다시 **[발주확정]** 버튼을 클릭하여 **`proc_fg: '4' ➔ '3'`**으로 전이시키고 매입전표를 새로 생성할 수 있습니다.

## 4. 핵심 데이터 제약 규칙 (FAQ)

### Q1. 매장 화면(st_vendor_00001)에서 확정 버튼이 주석 처리되어 보이지 않는 것은 버그인가요?
* **아닙니다. 정상 설계입니다.**
* 가맹점(매장)이 독자적으로 발주 요청을 본사 승인 대기 상태(`'5'`)로 확정하는 것을 예방하고, 본사 관리자가 `구매의뢰요청관리(hq_vendor_00001)` 화면에서 각 매장의 임시 저장 데이터(`'0'`)들을 취합하여 안전하게 직접 일괄 확정(`'5'`)하도록 프로세스가 설계 제어되어 있습니다.

### Q2. 이미 입고가 끝난 전표도 이 화면에서 발주취소나 수정이 가능한가요?
* **불가능합니다.** 
* 이미 매장에서 입고 검수를 완료하여 매입발주(`OBSLPHTB`)의 상태가 **`'4'` (입고확정)**가 된 전표는 본 화면(`hq_vendor_00002`)의 SQL 조회 조건(`WHERE A.OBSL_PROC_FG < 2 OR A.OBSL_PROC_FG IS NULL`)에 의해 **목록에서 아예 제외**되므로, 중복 처리가 철저하게 방지됩니다.

### Q3. 취소 흐름을 통해 구매의뢰를 아예 처음 매장 등록 상태로 되돌릴 수는 없나요?
* **정상 화면 UI 상에서는 지원하지 않습니다.** 
* 발주취소(`'4'`) 상태가 된 전표는 다시 재발주(`'3'`)하는 흐름만 가능하며, 다시 품의취소(`'2'`)나 의뢰요청(`'5'`) 상태로 직접 역행할 수 없습니다. 
* 만약 테스트 목적으로 강제 복원해야 하는 경우, 아래와 같이 DB에서 직접 SQL 업데이트를 통해 상태값을 조정해 주어야 합니다.

```sql
-- 구매의뢰 전표 상태를 '품의처리(1)'로 강제 원복한 후 화면에서 [품의취소]를 진행합니다.
UPDATE hmsfns.OBREQHTB 
   SET proc_fg = '1' 
 WHERE purch_req_no = '240201000701';

COMMIT;
```

### Q4. 구매의뢰등록(st_vendor_00001)에서는 조회가 되던 전표/상품이 본사 발주품의작성(hq_vendor_00002) 화면에선 왜 안 보이나요?
* **거래처 마스터 테이블(TVNDRMTB)과의 INNER JOIN 검증 유무 차이 때문입니다.**
  * **st_vendor_00001 (매장)**: 구매의뢰 상세 데이터(`OBREQDTB`)를 가져올 때 거래처 마스터와 조인하지 않고 순수 전표 테이블만 읽으므로 거래처 존재 여부와 무관하게 무조건 조회됩니다.
  * **hq_vendor_00002 (본사)**: 조회 쿼리(`getReqList`) 내부에서 거래처 명(`VN.VENDOR_NM`) 매핑 및 검증을 위해 **거래처 마스터 테이블(`hmsfns.TVNDRMTB VN`)과 `INNER JOIN`**을 맺고 있습니다 (`DT.PURCH_CONF_VENDOR = VN.VENDOR` 및 `MB.CHAIN_NO = VN.CHAIN_NO`).
* **결과**: `OBREQDTB`에 입력된 확정거래처(`PURCH_CONF_VENDOR`) 코드가 거래처 마스터(`TVNDRMTB`)에 없는 레거시 오류 데이터이거나, 프랜차이즈 체인번호(`CHAIN_NO`) 정보가 일치하지 않는 경우 **이너 조인 조건에 의해 해당 상세 행과 전표 자체가 조회 결과에서 완전히 누락**됩니다.
* **대처 가이드 (DB 데이터 정정 방법)**:
  - 이는 5년간 유지되어 온 고유의 정합성 검증 설계 스펙이므로 소스 코드 변경은 진행하지 않으며, 누락 발견 시 DB 레벨에서 아래 가이드를 참고하여 확정거래처 정보를 마스터에 존재하는 거래처 코드로 직접 수정해 주어야 합니다.
  - **대상 테이블:** `hmsfns.OBREQDTB` (구매의뢰 상세 테이블)
  - **대상 컬럼:** `purch_conf_vendor` (확정거래처), `purch_bas_vendor` (기본거래처)
  - **보정 SQL:**
    ```sql
    -- 거래처 정보 불일치로 누락된 전표(예: 260617000701)를 마스터에 존재하는 정상 거래처('000001')로 보정
    UPDATE hmsfns.OBREQDTB 
       SET purch_conf_vendor = '000001',
           purch_bas_vendor  = '000001'
     WHERE purch_req_no = '260617000701';

    COMMIT;
    ```


### Q5. 상품 추가 시 공급 거래처(기본/확정거래처)는 어떤 기준으로 자동 세팅되나요?
* 상품 등록 시점(`St_Vendor_00001_Mapper.addRequestGoods`)에 실행되는 거래처 자동 추적 로직은 **3단계 우선순위**에 따라 거래처를 매칭하고, 이 값을 **기본거래처(purch_bas_vendor)**와 **확정거래처(purch_conf_vendor)**에 동일하게 기록합니다.
  1. **1순위: 견적 정보 (`TESFRHTB`, `TESVDUTB`)**
     - 체인번호(`chainNo`)와 상품코드(`goodsCd`) 기준, 현재 요청일자가 견적 유효기간 내에 있고 견적단가 적용 플래그(`ESTIM_PRC_APPLY_YN`)가 `'Y'`인 거래처를 가장 우선적으로 채택합니다.
  2. **2순위: 최근 발주 내역 (`OBSLPHTB`, `OBSLPDTB`)**
     - 유효한 견적 정보가 없을 경우, 해당 가맹점(`msNo`)에서 이전에 해당 상품을 발주했던 최근 거래 내역을 역순 정렬하여 가장 최근에 거래한 발주 거래처를 가져옵니다.
  3. **3순위: 거래선 취급 상품 마스터 (`TVNDRGTB`)**
     - 견적 정보와 발주 내역이 모두 없는 경우, 거래선별 취급 상품 마스터 테이블에서 주거래처 플래그(`MAIN_VND_FG = '0'`)가 설정된 거래처를 기본값으로 채택합니다.
